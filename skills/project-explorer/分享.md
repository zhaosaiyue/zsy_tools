# 用 Claude Code Skill 10分钟读懂一个陌生项目

> 接手新项目第一天，你在干什么？
> 找人问架构、翻 Wiki、grep 代码、猜表结构……
> 等你把这些信息拼凑完，半天过去了。
>
> 现在有个更快的方式——让 AI 替你把项目从头扫一遍，生成一份完整的项目概览报告，10分钟内建立全局认知。

---

## 背景：测试同学接手项目有多痛

我们日常绕不开一个场景：**接手一个没文档、没人讲的存量项目**。

开发说"代码即文档"，但测试要写用例时：

- 接口在哪？方法、路径、入参是什么？——翻代码
- 表结构是什么样的？哪些字段必填？——问开发或者自己看 SQL
- 这个模块依赖了哪些中间件？——跑起来踩坑
- 有没有调用外部服务？调用的是什么接口？——不知道

这些信息散落在代码里，靠手动翻费时费力，靠问开发要等人且容易遗漏。

所以我做了一个 Claude Code Skill：**project-explorer**，专门解决"快速读懂陌生项目"这件事。

---

## 它能产出什么

一份结构化的 Markdown 项目概览报告，自动保存成文件，可以直接丢进知识库或贴给团队。

报告涵盖十个部分：

| 章节 | 内容 |
|------|------|
| 技术栈 & 依赖 | 语言/框架/ORM/MQ/缓存，自动从 pom.xml、package.json、go.mod 等读取 |
| 项目结构 | 目录树（3层）+ 每个模块的职责说明 |
| API 接口列表 | HTTP 方法 + 路径 + 入参 + 功能描述，逐文件扫描不遗漏 |
| HTTP 外部调用 | 项目主动发起的第三方/微服务调用 |
| 数据模型 & 数据库 | 表名、字段、类型、约束、关联关系 |
| 配置 & 环境变量 | 中间件地址、业务开关、.env 变量清单（敏感值自动过滤） |
| 核心业务逻辑摘要 | 每个 Service 的职责、外部依赖、关键判断逻辑 |
| 代码规范 & 工程实践 | 测试框架、CI/CD、日志、异常处理机制 |
| 待核查项 | 无法从代码确认的内容，明确标注，不瞎猜 |
| 快速上手建议 | 启动命令、必填配置、核心入口文件、高频接口 |

## 怎么用

Claude Code 里直接说触发词就行，不用记命令：

```
项目分析 / 读项目 / 项目概览 / 探索项目
```

AI 会弹出问题收集两件事：**项目路径** 和 **报告存哪**。填完自动跑，全程播报进度。

**三个常用快捷指令：**

| 指令 | 场景 |
|------|------|
| `全自动` | 两个阶段全部自动跑完，不问任何问题，适合中小项目 |
| `只要快照` | 只扫目录和技术栈，不深读代码，1分钟出结果，适合快速摸底 |
| `深入 支付模块` | Phase 2 只精读指定模块，大项目省时间，适合有明确分析目标时 |

## 它是怎么工作的

分两个阶段，可以分开执行：

**Phase 1 — 广度扫描**

用 `find` / `ls` 批量扫，不逐行读代码。自动识别技术栈文件、统计接口文件数量、模型文件数量、测试文件数量。速度很快，通常在几十秒内完成，跑完给你一个全局快照。

```
📸 Phase 1 快照完成
项目：order-service
语言/框架：Java + Spring Boot + MyBatis
模块数量：6 个（order / payment / notify / config / common / gateway）
接口文件：23 个 / 数据模型文件：18 个 / 测试文件：41 个
```

看完快照可以决定：要不要继续深入、重点看哪个模块。

**Phase 2 — 深度精读**

逐个读路由文件提接口、读 Model 文件提表结构、读 Service 提业务逻辑。**读完一个文件立即把结果写进报告**，实时播报进度，不会攒完所有内容才生成——大项目也不会卡死。

```
📖 正在读取：OrderController.java
✓ 已写入：提取到 8 个接口
📖 正在读取：OrderMapper.java
✓ 已写入：提取到 3 张表（order / order_item / order_log）
...
✅ 项目概览报告已生成
```

## 对测试同学的具体价值

**接手存量项目**
不用靠口口相传，跑一遍 10 分钟建立项目全貌，新同学第一天就能独立上手。

**写测试用例前**
接口列表和入参一次性扫出来，不用挨个找开发要文档，也不会遗漏边缘接口。

**构造测试数据**
数据表结构清单在手，哪些字段必填、有哪些外键约束、默认值是什么，一眼看清。

**排查线上问题**
快速定位该模块依赖了哪些中间件、调用了哪些外部服务，缩短排查时间。

**评估测试范围**
HTTP 外部调用章节直接列出所有第三方依赖，评估影响面时不会漏掉上下游。

## Skill 是什么，为什么值得自己做

Claude Code 的 Skill 本质上是一份写得很细的"任务说明书"——一个 Markdown 文件，告诉 AI 遇到特定触发词时，按什么步骤、用什么工具、生成什么格式来完成任务。

**核心价值：把经验固化成可复用的流程。**

写好一次，之后每次接手新项目直接复用，不用重新想步骤，不用重新写 Prompt，不用担心每次 AI 理解偏差。团队里其他人也可以直接用，经验就共享出去了。

对测试同学来说，很多重复性的分析工作都可以这样沉淀——测试用例生成、接口对比、测试数据清理……每个方向写一个 Skill，积累下来就是团队的效率资产。

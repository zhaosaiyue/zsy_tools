# 测试用例模板（XMind 风格）

> 来源：分析 /Users/zsy/Desktop/case/ 下 64 个 XMind 文件归纳，覆盖业绩、财务、退款、分账、监管等多个方向

---

## 一、整体结构

```
{需求名称}                              ← 根节点（与需求标题一致）
├── 文档                                ← 【固定，放首位】
│   └── ...
├── 测试物料                             ← 【按需，有则放文档后面】
│   └── ...
├── {功能模块 / 测试分类1}               ← 一级节点
│   ├── {场景描述}                       ← 二级节点（一个 case）
│   │   ├── {预期结果 / 分支}            ← 三级节点
│   │   │   └── {补充说明 / 数据验证}    ← 四级节点（可选）
│   │   └── {异常分支预期}
│   └── {场景描述2}
├── {功能模块2}
│   └── ...
└── 回归                                 ← 【固定，放末位】
    └── ...
```

---

## 二、节点详细说明

### 根节点
- 直接使用需求名称，与 PRD 标题保持一致
- 示例：`财务手机app审核退费单`、`退款流程改造`、`合作提报支持定软电业绩`

---

### 文档节点（固定，放首位）

**基础格式（必填）：**
```
文档
├── ones / keones：{ONES需求链接}
├── prd / PRD：{PRD文档链接}
└── 技术文档：{技术方案链接}
```

**扩展字段（按需添加）：**
```
文档
├── ones：{链接}
├── prd：{链接}
├── 技术文档：{链接}
├── 交互稿：{Figma链接}           ← 有前端交互时加
```
---

### 测试物料节点（按需，放文档后）

**适用场景：** 后端逻辑复杂、需要接口/数据库/账号辅助验证时

```
测试物料
├── 接口
│   ├── {接口用途}
│   │   ├── {接口路径}
│   │   ├── 入参：{关键入参说明}
│   │   └── 出参：{关键出参说明}
│   └── {接口用途2}
│       └── {接口路径}
├── 数据库
│   ├── {表名}：{用途说明}
│   └── {表名}：{用途说明}
├── SQL
│   └── {SQL语句或查询说明}
├── 账号
│   └── {角色/权限}：{账号信息}
├── 定时任务
│   └── {任务名}：{说明}
└── 前端地址
    └── {页面名}：{URL}
```

> 不是所有用例都需要测试物料，纯前端交互需求一般不加，后端逻辑验证类加。

---

### 一级节点（功能/测试分类）

根据需求类型选用，常见分类：

| 一级节点名 | 适用场景 |
|-----------|---------|
| `场景` | 核心业务流程测试（最通用，各领域均可用） |
| `测试场景` | 同"场景"，部分文件用此命名 |
| `需求用例` | 本次需求新增内容的测试 |
| `{具体功能名}` | 如"退款申请页"、"审核页"、"角色业绩计算" |
| `配置类` / `规则配置` | 后台配置、规则维护类需求 |
| `导入校验` / `导入检验` | 数据导入类需求 |
| `接口改造` | 接口变更类需求（含出参/入参验证） |
| `待确认点` | 文档未明确的内容，需开发/产品确认 |
| `回归` | **必须有**，放末位 |

> 同一个文件中可以有多个一级节点，按功能模块划分，不需要统一到"场景"下。

---

### 二级节点（场景描述）

**核心写法：** `{前置条件/状态}，{触发事件/操作}`

**正向场景示例：**
- `链家-退款申请审批通过-自动付款`
- `租赁-即刻租合同，合同额5000，预验成功，应收A`
- `新分账天津德佑-原路退-退款申请审批通过-自动付款`
- `CT单首次计入，审核通过，关联硬装角色人员和正签时一致`

**异常场景示例：**
- `链家-退款审批通过-发票状态未通过`
- `即刻租合同，合同额=1000（等于下限，不满足条件），命中其他区间规则`
- `CT单首次计入时间跨月`
- `财务A提交审核，已被财务B操作审核`

**命名原则：**
1. 把区分条件写进节点名（城市、分账类型、合同类型、金额区间等）
2. 正向/异常在节点名中体现，不用括号标注
3. 对于多条件组合，用`-`或`，`连接，不换行

---

### 三级节点（预期结果）

直接写结论，不写操作步骤。

**常见写法：**
- `退款审核通过，生成实收退款`
- `不自动支付，新增日志`（后接四级节点写日志文案）
- `CT单触发重新计算，复用硬装比例`
- `导入拦截`
- `配置保存成功，条件生效，查询回显正确`

**包含数据验证时：**
```
正向计算：正签金额 * 分配比例
  数据验证
    v1：计入业绩、角色类型、正签金额、角色人、比例、组织等
    v2：冲销业绩、角色类型、-正签金额、角色人、比例、组织等
```

**包含文案时，直接写全文案：**
```
不自动支付，新增日志
  文案：2024-10-01 00:00:00 财务系统 发票校验不通过无法自动支付
```

---

### 四级节点（补充说明，可选）

用于以下情况：
- **数据库字段验证**：`v1：字段名1、字段名2...`
- **文案内容**：写全提示文案原文
- **SQL**：`select * from table where ...`
- **待确认说明**：`文案是？？？`、`本次不做`
- **数字计算过程**：`计算过程：100 * (20% * 60%) = 12`

---

### 回归节点（固定，放末位）

覆盖本次改动可能影响的已有功能，写法简洁。

```
回归
├── {回归场景}
│   └── {预期}
└── {回归场景2}
    └── {预期}
```

示例：
```
回归
├── 无CT单，合作提报通过/管理员修改成功
│   └── 正常计算
├── 原路退，鉴权通过
│   └── 正常提交
└── 新分账天津链家-原路退-申请审批通过
    └── 自动付款
```

```
│   ├── 数据库
│   │   └── {表名}：{用途}
│   └── 账号
│       └── {角色}：{账号}

│   ├── {正向场景描述}
│   │   └── {预期结果}
│   │       └── {数据验证/文案（可选）}

│   │   └── {预期结果}

│       └── {预期结果}
├── 待确认点                         ← 有则加
│   └── {待确认内容}？？？
└── 回归
    ├── {回归场景1}
    │   └── {预期}
    └── {回归场景2}
        └── {预期}
```

## 四、场景命名速查

不同领域的二级节点常见命名模式：

**财务/退款类：**
- `{城市品牌}-{退款类型}-{操作}-{结果}`
- 示例：`新分账天津德佑-非原路退-退款申请审批通过-自动付款`
- 示例：`链家-退款审批通过-发票状态未通过`

**配置/规则类：**
- `{场景号/条件}：{具体配置条件描述}`
- 示例：`场景1：新分账，品效商圈门店，触发应分，分给特定公司金额计算正确`
- 示例：`启用商圈精耕配置，满足商圈精耕比例`

**后端计算/逻辑类：**
- `{状态}，{触发条件}，{关键参数}`
- 示例：`CT单首次计入，审核通过，关联硬装角色人员和正签时一致`
- 示例：`首次计入在7月，特殊变更在8月，人员在配置条件内`

**导入/数据类：**
- `{必填字段}为空，导入拦截`
- `{枚举值}不在范围内，校验失败`
- `疑似重复导入数据，校验异常`

**前端交互类：**
- `{操作路径}，{触发条件}`
- 示例：`退款金额为0，一体化场景，提示文案正确`
- 示例：`点击审核，已被他人操作，提交失败`

## 五、节点写法对比

| ❌ 差写法 | ✅ 好写法 |
|----------|----------|
| `正常场景` | `新分账天津链家-原路退-申请审批通过-自动付款` |
| `异常场景` | `发票状态校验不通过，不自动支付` |
| `验证数据` | `v1：计入业绩、角色类型、正签金额、角色人、比例、组织等` |
| `回归测试` | `无CT单，合作提报通过，正常计算` |
| `边界测试` | `即刻租合同，合同额=1000（等于下限），命中其他区间规则` |
| `待确认` | `文案是？？？` 或 单独建`待确认点`节点 |
| `操作成功` | `退款申请审批通过，生成实收退款，状态变为已支付` |

## 六、可选节点说明

以下节点根据需求类型**按需使用**，不强制：

| 节点 | 何时加 |
|------|--------|
| `测试物料` | 后端逻辑复杂，需接口/DB/账号辅助 |
| `待确认点` | 有未明确的文档细节，需开发/产品确认 |
| `数据验证` | 需要验证数据库字段正确性 |
| `前置条件` | 场景节点自身无法表达清楚时，用子节点补充 |
| `历史刷数` / `数据构造` | 数据初始化、存量数据处理需求 |
